## @sentry/react-native — major version review

Current: 7.11.0 (pinned range `~7.11.0`)
Latest: 8.13.0

Sentry SDK 8 ships a rewritten architecture. Key breaking changes in the 7→8 migration:
- `Sentry.init()` options shape changed; `integrations` API reworked
- `Sentry.wrap()` removed; use `Sentry.wrap` or `withSentryRouting` from the new Expo Router
  integration (`@sentry/react-native/expo-router`)
- `Sentry.Native.crashedLastRun()` moved to `Sentry.crashedLastRun()`
- `beforeSend` event type changes; `hub` API removed
- New `Sentry.getClient()` replaces `getCurrentHub().getClient()`

**Impact on this project:** The app uses `@sentry/react-native` only in `src/app/_layout.tsx`.
Surface area is small. However, the Expo Router integration path changed in v8, which may
require updating how Sentry wraps the root layout.

**Recommendation:** HOLD until SDK 56 upgrade. Sentry SDK 8 has a confirmed Expo SDK 56 /
Expo Router 4 integration path. Upgrade Sentry atomically with the SDK upgrade.
If updating standalone on SDK 55, verify against the v8 migration guide first:
https://docs.sentry.io/platforms/react-native/migration/v7-to-v8/

## marked — major version drift

Current: 9.1.6
Latest: 18.0.5 (9 → 18, MAJOR)

Usage: `scripts/build-privacy.js` only — a local Node.js build script, not bundled into the
app. This is a devDependency-class package used only at build time.

**Breaking changes (9→18):** The `marked` package ships multiple major versions per year.
Version 18 likely removes legacy extension APIs and changes renderer hooks. However, the
script uses only `marked(stripped)` — the simplest API form, which has been stable across
major versions.

**Recommendation:** LOW RISK. Can update independently. Run after updating:
```bash
npm install marked@18
node scripts/build-privacy.js  # smoke test
```
If the script fails, the old API is trivially restored.

## Security audit — npm audit

Total advisories: 14 moderate, 0 high, 0 critical.

All 14 moderate advisories trace back to the same root:

```
uuid (in @expo/config-plugins chain) — missing buffer bounds check in v3/v5/v6
```

The full chain: `uuid` → `xcode` → `@expo/config-plugins` → `@expo/config` → `expo`,
`@expo/cli`, `jest-expo`, `react-native-css`, `nativewind`, `@sentry/react-native`,
`expo-splash-screen`, `@expo/prebuild-config`, `@expo/metro-config`, `@expo/local-build-cache-provider`.

**Impact assessment:**
- The `uuid` vulnerability affects buffer-provided v3/v5/v6 UUID generation with a custom
  `buf` argument. The `xcode` package (a build-time dependency) calls `uuid` during Xcode
  project file manipulation — never at runtime on device.
- There is no runtime exposure in the shipped app. No user-facing code path calls uuid.
- Fixing requires Expo to update `@expo/config-plugins` to use a patched uuid version.
  This is outside project control and will resolve in SDK 56.

**Recommendation:** ACCEPT for now. No action required. Will resolve with SDK upgrade.
